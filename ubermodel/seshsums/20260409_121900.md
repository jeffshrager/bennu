# Session Summary — 2026-04-09

## What we worked on

Refactored `ubermodel.py`, a photo-catalytic methane destruction cost model
originally translated from `Cost_of_methane_destruction.xlsx`.  The script had
been a flat, single-execution Python file with no CLI, no parameter separation,
and no output persistence.  This session restructured it into a proper
command-line tool with named scenario settings, parameter sweeps, and
timestamped TSV output.

---

## Changes made

### 1. Separated parameters into three tiers

The original script mixed physical constants, scenario-specific numbers, and
derived calculations in one flat namespace.  These are now clearly separated:

**`PRIMARY_INPUTS` (dict)** — physical constants and experimentally-measured
values that are independent of deployment scenario:
- `ch4_per_kwy` — core conversion ratio (t CH₄ per kWy), the key experimental result
- `co2e_ratio` — 100-yr GWP of CH₄ (28)
- `ch4_reference_concentration_ppb` — concentration at which the ratio was measured
- `credit_price_per_tonne_ch4` / `credit_price_per_tonne_co2e` — market prices

**`SETTINGS` (dict of named dicts)** — scenario-specific engineering and economic
parameters, one bundle per deployment context:
- `ship` — shipboard device parameters (power, days at sea, fleet size, electricity cost)
- `solar` — ground-mount solar field parameters (nameplate MW, capex, capacity factor, lifetime)
- `landfill` — landfill solar parameters (construction premium, local CH₄ concentration,
  plus shared solar-field params)
- `troposphere` — global-scale parameters (troposphere volume, nuclear plant output,
  LED multiplier, campaign horizon)
- `all` — merged union of all four bundles (default)

**Computed outputs** — returned as named dicts from four `compute_*()` functions,
never stored as module-level variables.

### 2. Named scenario settings + `--setting` CLI flag

```bash
python3 ubermodel.py --setting solar
python3 ubermodel.py --setting landfill
python3 ubermodel.py --setting all       # default
```

### 3. Per-parameter CLI overrides with range sweep support

Any parameter (primary input or scenario setting) can be overridden from the
command line.  Values can be a single number or a `[low,high,step]` range:

```bash
python3 ubermodel.py --setting all --ch4_per_kwy 4.5
python3 ubermodel.py --setting solar --solar_nameplate_mw '[50,200,50]'
python3 ubermodel.py --setting all --ch4_per_kwy '[1,5,1]' --co2e_ratio '[20,30,5]'
```

Multiple range parameters produce a Cartesian product — one output row per
combination.  Shell users must quote range strings to prevent glob expansion.

### 4. Timestamped TSV output in `results/`

Every run writes a tab-separated file:

```
results/results_<setting>_<YYYYMMDD_HHMMSS>.tsv
```

Columns: `setting` | all parameter keys (in setting order) | all computed output keys.
When a range sweep produces multiple combinations, each combination is one row,
with its specific parameter values recorded alongside the outputs.  This makes
the file self-documenting and directly importable into spreadsheets or pandas.

### 5. Bug fix: solar and landfill cost calculations

The original script computed `solar_net_ch4`, `solar_cost_per_tonne_ch4`, and
the equivalent landfill metrics using `ship_energy_cost_per_year` — a variable
from the ship scenario that happened to be in scope.  This was clearly a
cross-scenario contamination bug (solar energy has no fuel cost; the appropriate
cost figure is the amortized annual capital expenditure).

Fixed: each scenario now computes its own annual cost basis.
- Ship: `ship_energy_cost_per_year = ship_energy_used_kwy × energy_cost_ship_per_kwy`
- Solar: `solar_annual_capex = (nameplate_mw × cap_cost_per_mw) / lifetime_years`
- Landfill: same formula multiplied by `landfill_construction_premium`

## Key design decisions

- **Flat parameter dict** passed into each `compute_*()` function rather than
  using a class or dataclass.  Keeps the code simple and makes it easy to dump
  all params directly into a TSV row.
- **Cartesian product for multi-range sweeps** rather than parallel iteration.
  This is more useful for sensitivity analysis (explore the full parameter space)
  even though it can produce large output files for wide ranges.
- **`all` setting merges all scenario bundles**, so a single run covers every
  scenario.  Individual settings exist for focused sweeps where you don't want
  the other scenarios' parameter columns cluttering the TSV.
- **No numpy dependency** — range generation is done with a simple while-loop
  to keep the script dependency-free.
