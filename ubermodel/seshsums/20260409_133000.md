## Change 1: Three-dimensional scenario structure

The original `SETTINGS` dict conflated deployment context, power source, and
device technology into one flat bundle.  This session separated them.

### Previous structure
```
SETTINGS = { 'ship': {...}, 'solar': {...}, 'landfill': {...}, 'troposphere': {...} }
```
One flat dict per scenario.  `B_led_efficiency_multiplier` was stranded inside
`troposphere` even though it is a property of the device, not the location.

### New structure
```
DEPLOYMENT_SETTINGS  — ship, field, landfill, troposphere
POWER_SETTINGS       — solar, nuclear, ship_engine, grid
DEVICE_SETTINGS      — fluorescent, led
```

Three separate CLI flags:
```
--setting  ship | field | landfill | troposphere | all
--power    solar | nuclear | ship_engine | grid
--device   fluorescent | led
```

`--setting all` runs the four canonical combinations that reproduce the original
four scenarios, applying `--device` to all of them.  `--power` can override the
canonical power source (unsupported pairs are warned and skipped).

### Key design decisions

- `B_device_efficiency_multiplier` now lives in `DEVICE_SETTINGS` and applies
  universally: in forward models it multiplies `ch4_per_kwy`; in the troposphere
  inverse model it divides the energy needed.
- The old setting name `solar` was renamed to `field` (open ground / ambient)
  to avoid collision with the new power source name `solar`.
- Supported setting+power pairs are enumerated in `SUPPORTED_COMBOS`.
  Unsupported pairs warn and skip rather than crashing.
- `build_params()` merges PRIMARY_INPUTS + DEPLOYMENT + POWER + DEVICE dicts,
  then applies any CLI overrides.  The single merged dict `p` is passed to
  `compute()`, which dispatches on `(setting_name, power_name)`.
- The troposphere scenario is structurally different (inverse: energy needed →
  how many plants/MW?) from the forward models.  `compute()` handles this with
  an early `if setting_name == 'troposphere':` branch.
- Troposphere now outputs both `O_nukes_raw` (1-year baseline, no improvements)
  and `O_nukes_needed` (with device efficiency × engineering × campaign years),
  making the original hardcoded LED assumption explicit and user-controllable.

### New combination: troposphere + solar

The original model hardcoded nuclear for the troposphere scenario.  The new
structure naturally supports `--setting troposphere --power solar`, computing
the solar nameplate MW, acreage, and cost needed to remove 1 ppm from the
troposphere over the campaign.

## Change 2: `--expname`, output filename cleanup, TSV header

### `--expname`
New CLI argument.  Defaults to the run timestamp (`YYYYMMDD_HHMMSS`).
The output file is `results/<expname>.tsv` — no `results_` prefix (redundant
given the directory), no appended setting or timestamp in the filename.

Examples:
```
python3 ubermodel.py                          → results/20260409_131500.tsv
python3 ubermodel.py --expname baseline       → results/baseline.tsv
python3 ubermodel.py --expname led_sweep ...  → results/led_sweep.tsv
```

### TSV header block
Every output file now begins with `#`-prefixed human-readable lines before the
TSV data:
```
# expname:   led_sweep
# timestamp: 20260409_132409
# setting:   field
# power:     (default per setting)
# device:    led
# override:  B_solar_nameplate_mw = [50.0, 100.0, 150.0, 200.0]
# combos:    field+solar+led, field+solar+led, ...
# rows:      4
# cli:       --expname led_sweep  --setting field  --device led  --B_solar_nameplate_mw [50.0,200.0,...]
#
setting  power  device  I_...  B_...  P_...  O_...
```

This makes every file self-documenting.  The `#` lines can be stripped before
import into pandas/Excel, or many parsers support `comment='#'` directly.

### TSV column order
`setting`, `power`, `device` (identity) | sorted `I_` | sorted `B_` | sorted `P_` | sorted `O_`

Columns are the union across all rows in the file; cells are empty string where
a column does not apply to a given combination (e.g. `O_nukes_needed` is empty
for non-troposphere rows in an `--setting all` run).

## Change 3: `run.sh` test suite

`run.sh` covers nine test sections with 25 individual invocations:

| Section | What it tests |
|---|---|
| 1 | Default run (all canonical combos) |
| 2a–d | Each setting individually |
| 3a–b | Non-default power sources; unsupported-combo warning |
| 4a–c | LED device; all+led; troposphere+nuclear+led ~1.9e3 result |
| 5a–d | Single-value parameter overrides (I_ and B_) |
| 6a–e | Single-parameter range sweeps |
| 7a–c | Multi-parameter Cartesian-product sweeps (2-param 9 rows, 3-param 8 rows, all×2params 36 rows) |
| 8a–c | Named experiments including tundra solar deployment |
| 9 | Sanity-check row counts on key output files |

All sections completed without errors.  Section 3b correctly produced a warning
for `ship+solar` (unsupported) and ran the remaining three combos.
